A central secrets management platform,
aligned to your business domains.

An enterprise-grade, self-hosted platform built on OpenBao (the open-source Vault fork) and run on Kubernetes / OpenShift. It issues key/value and auto-rotating Active Directory credentials, injects them directly into GitLab CI/CD pipelines and OpenShift workloads, and isolates every team behind a vault mapped to its business domain — all managed as code.

What it does

Capabilities

Every credential is short-lived, job-scoped, or auto-rotated — nothing long-lived is ever copied into a repository, CI variable, or cluster manifest.

Key/Value secrets

Versioned kv-v2 store with one mount per team, full self-service through OpenBao's built-in UI behind corporate SSO.

AD auto-rotation

The LDAP secrets engine rotates Active Directory service-account passwords on a schedule, plus check-out/check-in for shared accounts.

GitLab CI/CD injection

Pipelines mint an OIDC id_token; OpenBao verifies it and returns a job-scoped token. Zero credentials stored in GitLab.

OpenShift injection

The Vault Secrets Operator syncs secrets into native Secret objects — workloads consume them with no app changes.

Domain-aligned tenancy

Each business domain gets its own namespace, mounts, and least-privilege policies — self-service without cross-team blast radius.

Governance & resilience

Dual audit devices to a SIEM, PKI for service TLS, rate/lease quotas, Prometheus alerting, and snapshot-based DR.

How it fits together

Architecture

Identities flow in through federated auth; secrets flow out to pipelines, clusters, and rotated directory accounts. The OpenBao core runs HA on integrated Raft storage with Transit auto-unseal.

The core idea

Mapping vaults to business domains

Instead of one shared vault, each business domain owns an isolated slice of the platform: its own namespace, its own KV mount and AD roles, and a least-privilege policy that is auto-attached to that domain's people, pipelines, and workloads. One change to a domain's definition propagates everywhere — humans, CI, and pods — with no hand-edited ACLs.

The mapping is data, not boilerplate: a domain is declared once in Terraform variables, and the KV mount, AD static roles, generated ldap-read-<domain> policy, OIDC group binding, GitLab project role, and Kubernetes workload role are all derived from it.

Secret delivery

Integrations

Two first-class delivery paths, both credential-free at rest.

GitLab CI/CD pipeline

Pipeline mints an id_token with the platform as its audience.

OpenBao verifies it against GitLab's OIDC discovery URL and matches a project-bound role.

A job-scoped token reads only that domain's secrets, then expires with the job.

 .gitlab-ci.yml
deploy:
  id_tokens:
    VAULT_ID_TOKEN: { aud: https://gitlab.example.com }
  secrets:
    DB_PASSWORD:
      vault:
        engine: { name: kv-v2, path: kv/payments }
        path: payments-api
        field: db_password
  script: [ ./deploy.sh ]   # DB_PASSWORD injected, never logged

OpenShift / Kubernetes

The workload's ServiceAccount authenticates via the Kubernetes auth method.

The Vault Secrets Operator materializes the secret into a native Secret.

The pod consumes it as env/volume; rotations re-sync automatically.

 VaultStaticSecret
apiVersion: secrets.hashicorp.com/v1beta1
kind: VaultStaticSecret
spec:
  type: kv-v2
  mount: kv/payments
  path: payments-api
  refreshAfter: 60s
  destination: { name: payments-api-db, create: true }
# pod: envFrom → secretRef → payments-api-db

Active Directory — automatic rotation

 bao CLI
# OpenBao owns the AD service account and rotates it on a schedule
bao read ldap/static-cred/payments-svc      # current username + password
bao write -f ldap/rotate-role/payments-svc   # force an immediate rotation

# Shared-account pool: borrow exclusively, auto-rotate on return
bao write ldap/library/payments-batch/check-out ttl=300
bao write ldap/library/payments-batch/check-in  service_account_names=svc-batch-1

Governance & resilience

Config as code

Mounts, policies, auth roles, and AD rotation all live in Terraform; cluster manifests in GitOps. Every change is a reviewed PR — that review is the approval workflow.

Audit → SIEM

Two audit devices (durable file + streaming socket) with fail-safe semantics, forwarded to the SIEM. HMAC'd values, never plaintext.

Least privilege

Per-domain policies scope access to a single mount. GitLab roles are pinned to a project; workload roles to a ServiceAccount + namespace.

Resilience & DR

HA on Raft with Transit auto-unseal, scheduled snapshots, and a rehearsed restore runbook with RPO/RTO targets.

Observability

Prometheus scrape with alerts for sealed nodes, lost leadership, audit failures, and missing metrics — plus a Grafana dashboard.

Defense in depth

Internal PKI for service TLS, request rate limits, lease-count quotas, and default-deny NetworkPolicies around the cluster.
